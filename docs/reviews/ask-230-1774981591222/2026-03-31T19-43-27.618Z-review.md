## Executive Summary

This branch adds an install request workflow to the Discord bot, allowing users to trigger package installations that are tracked in the database, executed through a registry of allowlisted installers, and whose resulting binaries/environment are injected into subsequent /ask executions. The core design (installer registry, scoped install roots, env merging) is sound and well-structured. However, both reviewers independently identified that the handleDelete and handleReview status guards only check for 'queued' and 'running', missing the new install-in-progress states ('install_approved', 'install_running'), which could allow branch deletion or review initiation while an install is actively using the worktree. This correctness bug warrants revision before merge.

## Blocking Issues

- handleDelete and handleReview status guards miss install-in-progress states (src/discord/bot.ts): Both guards (bot.ts:1243 and bot.ts:1818) only check for 'queued' and 'running' statuses. The new 'install_approved' and 'install_running' states bypass these checks, meaning a user could /delete a branch or /review a request while an install is actively using the worktree, risking data corruption or conflicting operations.

## Non-Blocking Issues

- updateInstallRequest uses overwrite semantics for nullable fields (src/db/database.ts): All nullable columns except approved_by_user_id use direct ? parameters without COALESCE. Current call sites are correct (the 'running' update hits a fresh row where those columns are already NULL, and the 'succeeded'/'failed' updates pass all fields), but the API shape is error-prone for future callers. Consider COALESCE for all nullable columns.
- Redundant ALTER TABLE migrations for install_requests (src/db/database.ts): The CREATE TABLE already defines all columns. The subsequent ALTER TABLE ADD COLUMN loop is dead code on any database initialized with this code version, with some default divergences (e.g. DEFAULT '' in ALTER vs no default in CREATE). Should be cleaned up before merge.
- install_requested status defined but never set in production code (src/db/types.ts): RequestStatus includes 'install_requested' but no production code path sets it; createApprovedInstallRequest jumps directly to 'install_approved'. Only test fixtures use it. Either remove the status or add a production code path.
- env snapshot always passed to execution services even with zero installs (src/discord/bot.ts): buildExecutionEnvironment always creates a { ...process.env } snapshot. Previously spawn() inherited live process.env. While no runtime process.env mutation was found post-startup, the behavioral change is unnecessary for the zero-installs case. Consider only passing env when packages.length > 0.
- VERIFY_FAILED error code defined but never thrown (src/services/installService.ts): The wrapper verification step has no specific try-catch, so failures propagate as raw errors without the VERIFY_FAILED code. The outer catch still marks the install as failed, but loses verification-specific context.

## Missing Tests

- No test for runInstall failure path (spawnCollect rejection during install step, verifying DB updated to 'failed')
- No test for wrapper verification failure propagation
- No test for handleInstall bot command (permission checks, scope validation, thread requirement)
- No test for unknown package or unsupported scope rejection in createApprovedInstallRequest

## Disputed Issues

- Severity of updateInstallRequest overwrite semantics: one reviewer called it blocking (misleading API with destructive defaults), the self-critique and second reviewer called it non-blocking (current call sites are correct, TypeScript signature makes semantics clear). Resolved as non-blocking.

## Blocking Issues

**1. `updateInstallRequest` nulls out fields on partial updates (database.ts:597-630)**

The `bin_path`, `env_json`, `logs`, `error_message`, and `completed_at` columns use direct `?` parameters without COALESCE. The "running" state update at `installService.ts:123-128` only passes `status`, `approvedByUserId`, and `binPath` — meaning `env_json`, `logs`, `error_message`, and `completed_at` are all set to NULL. This is acceptable only because the first update is always at the start of a fresh install where those columns are already NULL. However, if `updateInstallRequest` is ever called for a status transition after data has been written (e.g. retrying or updating a partially-completed install), it will silently erase stored data. Either use COALESCE for all nullable columns or document this as intentional overwrite-all semantics. Promoting to blocking because the API shape is misleading — optional-looking parameters have destructive defaults.

**2. Redundant and confusing migration columns (database.ts:178-196)**

The `CREATE TABLE IF NOT EXISTS install_requests` block already defines all columns (`thread_id`, `package_version`, `approved_by_user_id`, `install_root`, `bin_path`, `env_json`, `logs`, `error_message`, `completed_at`). The migration block then tries to `ALTER TABLE ... ADD COLUMN` for these same columns. On a fresh database, the CREATE TABLE creates them all, and every ALTER fails (caught silently). On an upgraded database that had the table without these columns, the ALTERs would add them. But the CREATE TABLE already has them, so the ALTERs can never succeed on any database initialized with this code version. This is dead code on any new install. Worse, some of the ALTER defaults differ from the CREATE TABLE definitions (e.g. `package_version TEXT NOT NULL DEFAULT ''` in the ALTER vs `package_version TEXT NOT NULL` in CREATE TABLE). This suggests the migration was copied from an iterative development process without cleanup. Not a runtime bug, but confusing and should be reconciled before merge.

## Non-Blocking Issues

**1. `env` is always passed to child processes even with zero installs (bot.ts:2008-2011)**

`buildExecutionEnvironment` always returns `{ ...process.env }` (a snapshot). This is passed unconditionally to every `/ask` execution. Before this change, `spawnCollect` was called without an explicit `env`, which means `spawn()` inherited the live `process.env`. Now every request gets a frozen snapshot. This is functionally equivalent for the current code, but it's a behavioral change for all requests, not just those with installs. Consider only passing `env` when installs actually exist.

**2. `runQueuedGuildTask` wraps the request queue but doesn't create a request record (bot.ts:1415-1424)**

The install runs through the same `requestQueue` as `/ask` requests, consuming one of the bounded concurrency slots. This is probably fine for fairness, but there's no visibility — installs don't appear in the queue state logging since `requestQueue` fires `onStateChange` without knowing it's an install vs ask. Consider logging install queue entry separately.

**3. Request status pollution (types.ts:37-44)**

`RequestStatus` now includes `install_requested`, `install_approved`, `install_running`, `install_succeeded`, `install_failed`. These are only relevant when an install is linked to a request, but any code filtering or displaying request statuses (e.g. a `/repos` summary showing request counts by status) must now handle these new states. Check all consumers of `RequestStatus`.

**4. Wrapper verify step has no error handling (installService.ts:160-175)**

The wrapper verification (`spawnCollect(wrapperPath, wrapper.verifyArgs, ...)`) is outside the per-step try/catch. If verification fails, the raw error propagates without being wrapped in `InstallServiceError` with a `VERIFY_FAILED` code (which exists but is never used). The outer catch will still mark it failed, but the error message won't indicate it was a verification failure.

**5. `shellQuote` doesn't handle all edge cases (installerRegistry.ts:35-37)**

The `shellQuote` function handles single quotes but not other special characters. Since it wraps in single quotes this is technically correct for bash, but the install root path comes from config + user-controlled thread IDs. Discord thread IDs are numeric snowflakes so this is safe in practice, but worth a comment.

## Missing Tests

1. **No test for `runInstall` failure path** — The happy path is tested but there's no test where `spawnCollect` rejects during an install step, verifying the DB is updated to `failed` and the error propagates correctly.
2. **No test for `handleInstall` bot command** — `botCommands.test.ts` doesn't test the `/install` command handler (permission checks, scope validation, thread requirement for request scope).
3. **No test for `buildExecutionEnvironment` with zero installs** — Should verify it returns empty packages/pathEntries and a clean env.
4. **No test for unknown package rejection** in `createApprovedInstallRequest`.
5. **No test for unsupported scope rejection** (e.g. if a package only supports `repo` but `request` is requested).

## Strong Concerns

The `env` passthrough to execution services is the area most likely to cause a subtle regression. By always passing `{ ...process.env }` as a snapshot, any code that modifies `process.env` after bot startup (e.g. for Gemini API key rotation) won't be reflected in spawned processes. The previous behavior of omitting `env` (inheriting live `process.env`) was safer for the zero-installs case. The fix is simple: only pass `env` when `packages.length > 0`.

The overall design is sound — allowlisted packages, scoped install roots, wrapper scripts, and env merging are all reasonable approaches. The installer registry is extensible without being over-engineered.

## Confidence

**Medium-high.** The core install flow is well-structured and the intent is clear. The blocking issues are real but straightforward to fix. The env passthrough concern is the most impactful potential regression. Test coverage for the new service is decent but the bot command handler and failure paths need coverage.
