## Executive Summary

This branch moves provider CLI installations (claude, codex, gemini) from Docker image build time into the container entrypoint, seeding them into the persistent /data volume so they survive restarts without requiring image rebuilds. All three reviewers found no happy-path correctness defects — the permission model (USER appuser set before ENTRYPOINT) is correct, PATH wiring is coherent, and the README update accurately reflects the new operator workflow. The primary risks are operational: first-boot and repair installs now depend on npm registry availability, and because entrypoint.sh uses set -eu with a restart: unless-stopped policy, a transient registry outage produces a hard restart loop rather than the prior behavior of degrading individual /ask requests. Version resolution also moved from image-build time to container-start time, meaning the same image tag may ship different CLI versions across deployments. These are deliberate architectural tradeoffs rather than implementation defects, and consensus holds that no issue rises to blocking.

## Changed Files

- Dockerfile
- README.md
- docker/entrypoint.sh

## Blocking Issues

- None

## Non-Blocking Issues

- Redundant PATH export in entrypoint (docker/entrypoint.sh): The Dockerfile already sets PATH=/data/home/appuser/.npm-global/bin:$PATH via ENV. The export PATH="$NPM_CONFIG_PREFIX/bin:$PATH" line in entrypoint.sh prepends the identical directory a second time on every start. Harmless but noisy, and obscures that the Dockerfile is the authoritative source.
- All-or-nothing reinstall logic instead of per-CLI checks (docker/entrypoint.sh): If any one of the three binaries is absent, the script reinstalls all three at @latest, overwriting versions an operator may have independently updated via docker exec. The README explicitly documents per-CLI manual updates, making this a real operator-facing behavior mismatch.
- No version pins on runtime npm installs (docker/entrypoint.sh): npm install -g ...@latest now runs at container startup rather than image build time. The same image tag may install different CLI versions on different hosts or dates, and a breaking upstream CLI release can affect any fresh deployment. The prior build-time install also lacked pins, but the risk surface is larger now because version resolution happens per-container-start rather than per-image-build.

## Missing Tests

- Conditional install logic in entrypoint.sh: all binaries present → skip install; any binary missing → install all three
- npm install failure path: what happens when the registry is unreachable (set -eu causes exit, restart: unless-stopped produces a restart loop — this should be documented and ideally verified)

## Disputed Issues

- Claude raised a concern that the HEALTHCHECK start-period=20s could cause Docker to kill the container during a long first-boot npm install. Gemini and the round-1 critique rebutted this: the actual HEALTHCHECK command is `node -e "process.exit(0)"`, which returns immediately regardless of entrypoint progress and does not verify application readiness. Docker will not kill the container on this basis. The underlying startup-latency observation is real but the Docker-kill mechanism does not apply.

## Outstanding Concerns

- Container restart loop on network failure: set -eu in entrypoint.sh turns a failed npm install -g into a non-zero PID 1 exit. Combined with restart: unless-stopped in docker-compose.yml and scripts/redeploy.sh, a transient npm registry outage during first boot or a repair cycle causes the service to loop rather than start degraded. Previously, provider CLIs were baked into the image and this failure mode did not exist. The change does not document this risk or provide a fallback (e.g., warn and continue if install fails rather than aborting).
- Runtime version drift as an operational regression: moving npm install from image build time to container startup means reproducibility and rollback confidence are weaker — the same actuarius:latest tag may behave differently on a new host depending on what the npm registry serves at boot time.

None. **Strong Concerns**

With a typical `restart: unless-stopped` or `always` policy this produces a hard restart loop until the registry is reachable. The old behavior (CLIs in image) meant the container always started cleanly and only failed at request-execution time. This is a meaningful operational regression: a transient npm registry outage during a container restart now takes the whole service down rather than degrading individual `/ask` requests. The change does not document this risk, and there is no fallback (e.g., skip install on network failure and start anyway, logging a warning).

The success criteria says "make the container start up with the required provider CLIs available" but does not address what should happen when they can't be fetched. Given that the intent is production use on GCP with a persistent volume, this scenario (container restarting after a deploy while npm registry is briefly unavailable) is realistic.

---

**Confidence**

Medium-high. The permission model is correct (USER appuser is set before ENTRYPOINT), the PATH setup is coherent, and the documentation additions are accurate. The restart-loop concern is real and worth addressing before this ships to production, but nothing in the implementation is outright broken under happy-path conditions.

### Gemini (gemini / gemini-3.1-pro-preview)

```text
Blocking Issues

None. The core mechanism correctly moves the provider CLIs out of the Docker image and seeds them into the persistent volume, maintaining their availability in the PATH without requiring image rebuilds.

Non-Blocking Issues

1. **Grouped installation condition (`docker/entrypoint.sh`)**: The script checks if *any* of the three CLIs are missing and, if so, installs *all three* using `@latest`. If a user manually uninstalls or deletes just one CLI, the next container restart will reinstall all three, clobbering any specific version pins the user might have manually applied to the other two via `docker exec`. The script should check and install each CLI individually.
2. **Redundant `PATH` manipulation**: The `Dockerfile` correctly sets `ENV PATH=/data/home/appuser/.npm-global/bin:$PATH`. In `entrypoint.sh`, `export PATH="$NPM_CONFIG_PREFIX/bin:$PATH"` prepends the exact same directory again. This is harmless but sloppy.

Missing Tests

- The project appears to test shell script logic (e.g., `tests/redeployScript.test.ts`). There is no test verifying the `entrypoint.sh` seeding behavior, such as ensuring `npm install -g` is executed when the volume is fresh, and correctly skipped when the binaries are present.

Strong Concerns

1. **Non-deterministic fresh installs**: Previously, running `npm install -g` in the `Dockerfile` meant the CLI versions were baked into the immutable image tag at build time. Now, any fresh deployment (e.g., starting the container with a new/empty `/data` volume) will run `npm install -g` and pull the `@latest` versions at runtime. The same `actuarius:latest` image might deploy successfully today but fail on a new host tomorrow if Anthropic, OpenAI, or Google publish a breaking CLI update. If the goal is just decoupling updates, consider seeding specific pinned versions in `entrypoint.sh` instead of unbounded `@latest`, while still allowing the user to manually upgrade later.

Confidence

High. The implementation directly satisfies the stated intent and the use of `$NPM_CONFIG_PREFIX` successfully routes global installs into the persistent volume.
```
