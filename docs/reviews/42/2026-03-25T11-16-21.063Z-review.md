## Executive Summary

This branch moves CLI auth and home-directory state onto the persistent /data volume so credentials survive container replacement, adds GitHub App auth support, and introduces multi-provider execution services. The core persistence model is sound, but one concrete ownership bug in the local bootstrap path remains unresolved.

## Changed Files

- Dockerfile
- README.md
- docker/entrypoint.sh
- scripts/redeploy.sh
- scripts/start-local.ps1

## Blocking Issues

- start-local.ps1 leaves /data/home/appuser/.claude root-owned on fresh bootstrap (scripts/start-local.ps1): Line 66 creates the .claude directory as root (docker exec -u 0 mkdir -p), but line 68 only chowns the credentials file, not the directory itself. On a fresh local start without prior migration, .claude/ ends up root:root 755, so appuser cannot create or replace files there. This directly violates the success criterion that migrated/bootstrapped files be writable by the app user.

## Non-Blocking Issues

- README documents CLAUDE_CREDENTIALS_FILE / CLAUDE_CREDENTIALS_B64 but nothing reads them (README.md): These env vars appear only in README.md (lines 63-64, 147) and are not referenced in any runtime code or script. This is docs/runtime drift that will confuse operators.
- redeploy.sh silently ignores partial GitHub App config when GH_TOKEN is present (scripts/redeploy.sh): When GH_TOKEN is set alongside incomplete App creds (e.g. GITHUB_APP_ID without key), the validation short-circuits and the partial creds are silently dropped. No runtime failure occurs—the system falls back to GH_TOKEN—but the stated intent to reject invalid auth combinations is not met. This is a defensive-coding/usability gap.
- Hardcoded UID 1001 in three independent files (Dockerfile): Dockerfile, redeploy.sh, and start-local.ps1 all encode UID 1001 independently. Changing the UID in one file silently breaks the others. Consider documenting the coupling or extracting a shared constant.
- entrypoint.sh omits XDG_DATA_HOME from mkdir (docker/entrypoint.sh): The entrypoint creates $HOME, $XDG_CONFIG_HOME, $XDG_CACHE_HOME, and $XDG_STATE_HOME but skips $XDG_DATA_HOME (/data/home/appuser/.local/share), which is defined in the Dockerfile. Most tools create it on demand so this is unlikely to cause failures, but it's an inconsistency worth fixing.
- README scope inconsistency — implies only Codex/Gemini benefit from persisted home (README.md): README says 'keeps Codex and Gemini authentication' but moving HOME to /data persists all CLI state including Claude's. The docs understate the actual scope of the change.
- Recursive chown on every redeploy may be expensive (scripts/redeploy.sh): chown -R 1001:1001 "$DATA_ROOT/home" in redeploy.sh runs on every deploy and traverses all files. A non-recursive chown of just the top-level directories would suffice since subdirectories are created by appuser.

- No coverage for the redeploy.sh auth-matrix validation: GH_TOKEN only, full GitHub App with raw PEM, full with B64 PEM, and partial/invalid combos

- Whether migration copying .gitconfig causes stale config — the entrypoint's git config --global overwrites existing values on every start, so migrated values don't persist.

### Claude (claude / claude-opus-4-6)

```text
## Blocking Issues

1. **entrypoint.sh missing `chown` — container may fail on first boot or after volume wipe.** The Dockerfile creates `/data/home/appuser` and chowns it at build time, but `/data` is a mount point. At runtime the volume contents replace what was built. The entrypoint runs `mkdir -p` but never `chown`s the directories it creates. If the volume is fresh (or the host pre-creates dirs as root), the `appuser` process will own nothing under `$HOME` and `git config --global` (which writes `$HOME/.gitconfig`) will fail. `redeploy.sh` has the `chown` on the host side, but `entrypoint.sh` does not — so any deployment path that doesn't go through `redeploy.sh` (docker-compose, local start, manual `docker run`) hits this. Add `chown -R appuser:appuser "$HOME"` in the entrypoint (it already runs as root before the `exec`/`su` hand-off, presumably).

2. **entrypoint.sh skips `XDG_DATA_HOME`.** The `mkdir -p` line creates `$HOME`, `$XDG_CONFIG_HOME`, `$XDG_CACHE_HOME`, and `$XDG_STATE_HOME` — but `XDG_DATA_HOME` (`/data/home/appuser/.local/share`) is set in the Dockerfile and never created. Any tool that tries to write there (and doesn't create it itself) will fail.

3. **redeploy.sh GitHub App validation logic is wrong.** The old validation used `&&` (required all three to be *empty* simultaneously), which was wrong in the opposite direction. The new validation uses `||` correctly for requiring all three, but the outer guard is still broken: the condition `[ -z "$GH_TOKEN" ] && (all-app-creds-missing)` means if `GH_TOKEN` is set *and* App creds are partially set (e.g., `GITHUB_APP_ID` is set but key is missing), the script silently proceeds — then later the `EXTRA_ARGS` block will skip forwarding the partial App creds (because its own guard requires all three). This is arguably fine in practice (partial creds are just ignored), but it means you can deploy with `GH_TOKEN` plus a broken App config and never get warned. The stated intent says "reject invalid GitHub auth environment combinations" — this doesn't do that.

4. **`redeploy.sh` fetches `GITHUB_APP_PRIVATE_KEY` from instance metadata, which stores it as a single metadata value.** GCP instance metadata values cannot contain newlines in the way PEM keys require (they get flattened). This is why the B64 variant exists. Adding `get_meta "env-github-app-private-key"` will silently produce a corrupted key unless the operator base64-encodes it anyway, at which point it's just the B64 path with a different name. This new code path is likely unusable in GCP production and will cause confusing failures.

## Non-Blocking Issues

1. **Hardcoded UID 1001 in three places** (`Dockerfile`, `redeploy.sh`, `start-local.ps1`). If anyone changes the UID in the Dockerfile, the scripts silently break. Consider extracting this to a shared constant or at least documenting the coupling.

2. **`start-local.ps1` migration copies everything including potentially stale/conflicting gitconfig.** The `cp -a /legacy/. /target/home/appuser/` copies the entire old home, including `.gitconfig`, `.claude/`, and anything else. If the old home had a `.gitconfig` with a different `user.name`/`user.email`, those values will persist and override the entrypoint's `git config --global` calls (since the file already exists and `git config --global` writes to the same file). The migration should either be selective or the entrypoint should use `--replace-all` / overwrite unconditionally.

3. **`chown -R 1001:1001 "$DATA_ROOT/home"` in `redeploy.sh` is expensive on re-deploy.** If `/data/home` contains many files (cloned repos, caches), this recursive chown runs on every deploy. Consider `chown 1001:1001 "$DATA_ROOT/home" "$DATA_ROOT/home/appuser"` (non-recursive) since subdirectories are created by the app user anyway.

4. **No `XDG_RUNTIME_DIR` set.** Some tools expect it. Minor, but worth noting if you're already setting the other four XDG variables.

5. **README says "Codex and Gemini CLI auth"** in two places but the Dockerfile/entrypoint changes are generic (they move the entire home). The documentation implies only Codex/Gemini benefit, but Claude credentials also live under this path. The docs should be consistent with the actual scope.

## Missing Tests

- There are no tests for any of the changed files. These are all infrastructure/deployment scripts, which are inherently hard to unit test, so the absence is understandable. However:
  - The GitHub auth validation logic in `redeploy.sh` is now complex enough to warrant a small shell test (or at least a table of expected pass/fail combos in a comment).
  - The migration logic in `start-local.ps1` (sentinel-based one-time copy) has no test coverage and the idempotency guarantee is load-bearing.

## Strong Concerns

The `XDG_DATA_HOME` omission in the entrypoint `mkdir` (blocking issue #2) is a copy-paste bug that will cause runtime failures for any tool using the XDG base directory spec. This is the kind of bug that passes manual smoke testing (because many tools create the directory themselves) but fails unpredictably in production.

The GCP metadata + raw PEM key concern (blocking issue #4) suggests this code path was not tested in the actual production environment. If it was, I'd like to understand how newlines in the PEM key survive GCP metadata storage and retrieval.

## Confidence

Medium-high confidence in the blocking issues. The `chown` gap and `XDG_DATA_HOME` omission are straightforward reads of the diff. The GCP metadata/PEM concern is based on well-known GCP metadata limitations but could be wrong if the project uses a non-standard metadata encoding. The validation logic concern is a definite gap relative to the stated intent.
```
