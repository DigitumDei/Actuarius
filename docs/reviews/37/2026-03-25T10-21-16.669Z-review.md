## Executive Summary

The branch moves the container HOME under /data so provider CLI auth survives container replacement — a sound design change. The Dockerfile, entrypoint, and redeploy validation logic improvements are solid. However, the local legacy-volume migration path in start-local.ps1 has two compounding bugs (unconditional re-run overwrites newer auth; root-owned home directory) that will break existing local users on the second restart. These must be fixed before merge.

## Changed Files

- Dockerfile
- README.md
- docker/entrypoint.sh
- scripts/redeploy.sh
- scripts/start-local.ps1

## Blocking Issues

- Legacy volume migration overwrites newer auth on every restart (scripts/start-local.ps1): start-local.ps1:40-44 checks whether the actuarius_home volume exists and unconditionally copies its contents into /data/home/appuser on every launch. After initial migration, any auth refreshed inside the running container (Codex, Gemini, Claude) will be silently overwritten by stale legacy state on the next restart. Fix: write a sentinel file after migration and skip the copy when it exists, or remove/rename the legacy volume after successful migration.
- Local migration creates root-owned home directory (scripts/start-local.ps1): The alpine migration container at start-local.ps1:43 runs as root and writes into the actuarius_data volume with no follow-up chown. The runtime starts as appuser (Dockerfile:48) and entrypoint.sh:7 expects a writable HOME. If the target home did not already exist with correct ownership, provider CLIs will fail to write config/cache/state. Fix: add chown -R to the migration container command targeting the appuser UID.

## Non-Blocking Issues

- Hardcoded UID 1001 in redeploy.sh coupled to image internals (scripts/redeploy.sh): redeploy.sh:67 hardcodes chown -R 1001:1001 but the Dockerfile does not pin appuser to UID 1001 (useradd has no --uid flag). infra/startup.sh:18 also assumes 1001. If the base image changes, the persisted home becomes unwritable. Fix is trivial: add --uid 1001 to the useradd in the Dockerfile.
- redeploy.sh only supports GITHUB_APP_PRIVATE_KEY_B64, not raw GITHUB_APP_PRIVATE_KEY (scripts/redeploy.sh): README and src/config.ts both support either the raw or base64 form of the GitHub App private key, but redeploy.sh:25 only reads and forwards the B64 variant. Users following the README with a raw key will hit a silent deployment-path mismatch.
- CLAUDE_CREDENTIALS_FILE / CLAUDE_CREDENTIALS_B64 documented but not wired in production deploy (README.md): The README (lines 63, 147) newly documents these as supported auth options, but redeploy.sh:26 only fetches and passes CLAUDE_CODE_OAUTH_TOKEN. This is a branch-introduced docs/runtime mismatch — either wire the new options or remove them from the README.

## Missing Tests

- No test or scripted verification for migration idempotency: first run with actuarius_home present, second run after updating auth in /data/home/appuser, asserting newer data is not overwritten
- No ownership verification after migration/redeploy confirming /data/home/appuser and XDG subdirectories are writable by appuser
- No production-path test for the GitHub auth matrix (GH_TOKEN only, full GitHub App creds, partial GitHub App creds)

### Claude (claude / claude-opus-4-6)

```text
## Blocking Issues

**1. Legacy volume migration overwrites newer auth on every restart (`start-local.ps1:40-44`)**

The script checks whether the `actuarius_home` volume exists and, if so, copies its contents over the current `/data/home/appuser` directory. This runs on every launch, not just once. After the first migration, if Codex or Gemini CLI auth is refreshed inside the running container, the next restart will silently overwrite it with stale state from the legacy volume.

Fix: either delete/rename the legacy volume after a successful migration, or write a sentinel file (e.g., `/target/home/appuser/.migrated`) and skip the copy when it already exists.

**2. Local migration creates root-owned home directory (`start-local.ps1:43`)**

The `alpine` container runs as root. `mkdir -p /target/home/appuser && cp -a /legacy/. /target/home/appuser/` produces a tree owned by `root:root` inside the `actuarius_data` volume. There is no follow-up `chown`. When the actual container starts, `entrypoint.sh:7` runs `mkdir -p "$HOME" ...` as `appuser` (via `USER appuser` in the Dockerfile). If the parent directories are already root-owned, these `mkdir -p` calls succeed only for paths that already exist — but any new subdirectory creation will fail with permission denied. Provider CLIs writing to `$XDG_CONFIG_HOME`, `$XDG_CACHE_HOME`, etc. will also fail.

Fix: add `chown -R` to the migration container command, or run a post-migration `docker exec -u 0` to fix ownership, similar to what `redeploy.sh:67` does.

**3. Hardcoded UID 1001 in `redeploy.sh:67` is coupled to image internals**

`chown -R 1001:1001 "$DATA_ROOT/home"` assumes `appuser` has UID 1001. This happens to be true today because `node:22-bookworm-slim` already has `node` as UID 1000, so the next `useradd` gets 1001. But the Dockerfile does not pin the UID (`--uid 1001`), so any base image change or Dockerfile reordering silently breaks this. If the UIDs diverge, the production home directory becomes unwritable after redeploy.

Fix: pin the UID in the Dockerfile (`useradd --uid 1001 ...`) or extract it at deploy time (`docker run --rm $IMAGE id -u appuser`).

## Non-Blocking Issues

**4. `redeploy.sh` validation logic fix is correct but could be clearer**

The `&&` → `||` change on line 31-32 fixes a real bug: the old logic only rejected the config when *all three* app credentials were missing; partial configs (e.g., only `GITHUB_APP_ID` set) would silently pass validation and then fail at runtime. The fix is correct — with `||`, missing *any* of the three triggers the error. Good catch, but worth a comment in the script explaining the intent.

**5. `VOLUME ["/data"]` in Dockerfile:49 interacts poorly with the new home layout**

Docker's `VOLUME` directive causes `/data` to be an anonymous volume when no explicit `-v` is passed. The image-baked `/data/home/appuser` (with correct ownership from the `RUN chown`) gets copied into the anonymous volume on first run, which works. But this is a subtle behavior that breaks if someone later adds a named volume mount — the image layer's ownership is hidden. This is pre-existing behavior that's now riskier because more state lives under `/data`.

**6. README documents `CLAUDE_CREDENTIALS_FILE` / `CLAUDE_CREDENTIALS_B64` but runtime only wires `CLAUDE_CODE_OAUTH_TOKEN`**

Pre-existing issue, not introduced here, but worth cleaning up to avoid confusion now that auth persistence is getting a proper story.

## Missing Tests

There are no automated tests for any of the changed files (shell scripts, Dockerfile, PowerShell). This is understandable for infrastructure scripts, but the migration logic in `start-local.ps1` is complex enough that at minimum the idempotency behavior should be manually verified and documented in the PR description.

No test covers the entrypoint `mkdir -p` behavior when HOME directories don't exist on the mounted volume.

## Strong Concerns

The core idea — moving HOME under `/data` so provider CLI auth survives container replacement — is sound. The Dockerfile and entrypoint changes are clean. The `redeploy.sh` validation fix (`&&` → `||`) and the guard for passing partial GitHub App creds (line 49) are both correct improvements.

However, the local migration path has two compounding bugs (runs every time + wrong ownership) that will bite anyone using the legacy volume flow. These aren't theoretical: the migration path is the only way existing local users transition to the new layout, so it will be exercised exactly once per developer — and it will break on the second restart.

The hardcoded UID is a ticking time bomb that should be addressed now while the UID coupling is being explicitly introduced into `redeploy.sh`.

## Confidence

**Medium-high.** I read every changed file in full. The Dockerfile/entrypoint/redeploy changes are solid. The three blocking issues are concrete and reproducible — they aren't speculative. The validation logic fix is a genuine bugfix. Overall this is a well-motivated change with a few sharp edges in the migration/deployment scripts that need filing down before merge.
```
