## Executive Summary

This branch makes persisted CLI authentication and home-directory state reliable across container redeploys by moving HOME onto the persistent /data volume with explicit XDG variables, adding auth validation to redeploy scripts, and introducing comprehensive test coverage for the shell-script auth matrix. The core architecture is sound and the auth validation logic is correct. No demonstrated blocking regressions exist in the supported upgrade path.

## Non-Blocking Issues

- -HomeVolume parameter renamed without backward compatibility (scripts/start-local.ps1): scripts/start-local.ps1 renames -HomeVolume to -LegacyHomeVolume with no alias shim. Any existing saved commands or wrapper scripts using the old parameter name will break immediately. PowerShell's [Alias("HomeVolume")] attribute would preserve compatibility at zero cost.
- mkdir/chown mock logs created but never asserted in tests (tests/redeployScript.test.ts): redeployScript.test.ts stubs mkdir and chown and logs their invocations (lines 90-96), but no test asserts on those logs. Adding assertions that mkdir -p and chown -R are called with the correct paths before docker run would lock in the ownership-repair contract.
- Legacy migration container launches on every local start (scripts/start-local.ps1): When the legacy actuarius_home volume exists but migration is already done, the alpine container still starts, checks the sentinel, and exits on every invocation of start-local.ps1. Minor startup overhead that could be avoided by checking the sentinel on the data volume first or documenting that removing the legacy volume after migration skips this.
- chown -R recurses entire home tree on every production redeploy (scripts/redeploy.sh): scripts/redeploy.sh:94 runs chown -R 1001:1001 on the full home directory every deploy. Currently negligible, but worth a comment explaining why full recursion is intentional (legacy content may have wrong ownership) or adding a conditional guard.

## Missing Tests

- GH_TOKEN + complete GitHub App config provided simultaneously: the script accepts and forwards both, the runtime prefers App auth, but no test covers this combination
- No test coverage for start-local.ps1 migration logic (sentinel-gated copy, chown, renamed parameter, no-legacy-volume path)
- No assertion on mkdir/chown invocation order relative to docker run in redeploy tests, despite mock infrastructure being in place

## Disputed Issues

- Local upgrade ownership repair gap: Codex initially flagged that existing actuarius_data volumes might have wrong UID after the explicit UID 1001 pin. Both reviewers ultimately agreed the old implicit UID was also 1001 (node:22-bookworm-slim assigns 1001 to the first useradd), so no demonstrated regression exists. Codex withdrew the blocker. Claude classified it as blocking then reclassified to non-blocking/hardening. Final consensus: not a blocker for supported upgrade paths.
- Dual-auth ambiguity (GH_TOKEN + GitHub App together): Codex argued the script should reject mixed auth or document precedence explicitly. Claude argued the documented preferred/fallback model makes accepting both correct behavior. Consensus: this is a test coverage and documentation gap, not a behavioral defect.

## Outstanding Concerns

- Local start-local.ps1 does not perform unconditional ownership repair before container start, unlike production's redeploy.sh which always runs chown -R. While not a demonstrated regression (UIDs match), the asymmetry means local is less defensively hardened than production against edge cases like externally mutated volumes.
