## Executive Summary

The PR moves the app user's HOME onto the persistent /data volume so CLI auth state survives container rebuilds. The core design is sound, but the local migration script has two bugs (repeated overwrite of newer state, root-owned migrated directories) and the production deploy script hardcodes a UID that is not pinned in the Dockerfile.

## Changed Files

- Dockerfile
- README.md
- docker/entrypoint.sh
- scripts/redeploy.sh
- scripts/start-local.ps1

## Blocking Issues

- Legacy migration re-runs on every local start, overwriting newer auth state (scripts/start-local.ps1): start-local.ps1:40-43 checks whether the legacy volume exists but never removes it or records a sentinel. On every subsequent invocation, `cp -a /legacy/. /target/home/appuser/` overwrites files that may have been updated since the first migration, defeating the persistence goal.
- Legacy migration leaves /data/home/appuser root-owned, breaking runtime writes (scripts/start-local.ps1): The migration at start-local.ps1:43 runs inside an alpine container as root. `mkdir -p /target/home/appuser` creates the directory owned by root. The runtime container starts as appuser and the entrypoint (entrypoint.sh:7) calls `mkdir -p` for XDG subdirectories, which will fail with permission denied on a freshly migrated volume where the parent is root-owned.
- Hardcoded UID 1001 in redeploy.sh not pinned in Dockerfile (scripts/redeploy.sh): redeploy.sh:67 runs `chown -R 1001:1001` assuming appuser has UID 1001, but the Dockerfile (line 43) uses `useradd` without `--uid`, relying on implicit allocation. If the base image changes and appuser gets a different UID, the persisted home becomes unwritable. Either the Dockerfile should pin the UID or redeploy.sh should extract it from the image.

## Non-Blocking Issues

- README documents GITHUB_APP_PRIVATE_KEY (raw PEM) but redeploy.sh only reads _B64 (scripts/redeploy.sh): README line 54 says either GITHUB_APP_PRIVATE_KEY or GITHUB_APP_PRIVATE_KEY_B64 are valid, but redeploy.sh:25 only fetches the base64 form from GCP metadata. A user configuring the raw PEM in production would get silent auth failure.
- XDG_DATA_HOME not set alongside other XDG variables (Dockerfile): The Dockerfile explicitly sets XDG_CONFIG_HOME, XDG_CACHE_HOME, and XDG_STATE_HOME but omits XDG_DATA_HOME. The default resolves correctly under the new HOME so nothing breaks, but it is an inconsistency.

## Missing Tests

- Migration idempotency: verify that a second local start after updating auth in /data/home/appuser does not overwrite newer state with stale legacy data
- Ownership correctness: verify that /data/home/appuser and its XDG subdirectories are writable by appuser after both fresh start and post-migration scenarios

## Disputed Issues

- Whether chown -R on every deploy (redeploy.sh:67) is a performance concern — one reviewer flagged it, the other noted it is necessary for ownership repair and the directory is small in practice
- Whether shell/Docker deployment scripts warrant automated tests or manual verification is sufficient — reviewers disagreed on practicality

## Outstanding Concerns

- README advertises CLAUDE_CREDENTIALS_FILE / CLAUDE_CREDENTIALS_B64 options that neither redeploy.sh nor start-local.ps1 consume — unclear if these are handled elsewhere or are dead documentation
- The consolidation from two volumes (data + home) to one volume was not explicitly evaluated for drawbacks such as backup granularity or volume lifecycle independence

### Claude (claude / claude-opus-4-6)

```text
## Blocking Issues

**1. Legacy migration in `start-local.ps1` runs on every start, not just once (line 41-43)**

The migration block checks whether the legacy volume *exists* but never removes it or marks the migration as complete. On every subsequent `start-local.ps1` invocation, `cp -a /legacy/. /target/home/appuser/` will run again, overwriting files in the new location with stale data from the legacy volume. If a user authenticates with a new provider after the first migration, the next restart will clobber that state with the old snapshot.

Fix: either delete the legacy volume after successful migration (`docker volume rm $LegacyHomeVolume`), or check for a sentinel file in the target before copying.

**2. Hardcoded UID 1001 in `redeploy.sh` (line 67)**

`chown -R 1001:1001 "$DATA_ROOT/home"` assumes `appuser` gets UID 1001. This is true today only because `node:22-bookworm-slim` already has a `node` user at UID 1000, so `useradd` assigns the next available. If the base image ever changes (different Node major, different distro variant, or an upstream update adds another user), this silently breaks — the container's `appuser` can't write to its own home. The Dockerfile should either pin the UID explicitly (`useradd -u 1001`) or `redeploy.sh` should extract the UID from the image rather than hardcoding it.

## Non-Blocking Issues

**1. `chown -R` on every deploy is unnecessarily expensive (redeploy.sh:67)**

`chown -R 1001:1001 "$DATA_ROOT/home"` recurses the entire persisted home tree on every deploy. As CLI auth state, caches, and config accumulate, this grows. A targeted `chown` on just the top-level directory (without `-R`) plus `mkdir -p` would suffice — the entrypoint already creates subdirectories, and files written by the container will already have the correct ownership.

**2. `XDG_DATA_HOME` is not set**

The Dockerfile sets `XDG_CONFIG_HOME`, `XDG_CACHE_HOME`, and `XDG_STATE_HOME`, but omits `XDG_DATA_HOME`. The default (`$HOME/.local/share`) happens to resolve correctly under the new `HOME`, so nothing breaks, but for consistency and explicitness it should be included alongside the others.

**3. Entrypoint `mkdir -p` could fail on a raw bind-mount (entrypoint.sh:7)**

The entrypoint runs as `appuser` (via `USER appuser`). On production, `redeploy.sh` pre-creates and chowns the directories before the container starts, so this works. But if someone runs the container directly with a fresh bind-mount (e.g., `-v /some/new/path:/data`), `/data` will be owned by root and `mkdir -p` will fail. This is an edge case since the documented path is through the scripts, but a comment in the entrypoint noting the assumption would help.

**4. README mentions only Codex and Gemini**

The README additions (lines about "Codex and Gemini CLI auth") mention only two providers. The mechanism actually persists *all* home-directory auth (including Claude `.claude/` state, `gh` credentials, etc.). The wording should be broader or explicitly list all affected providers.

## Missing Tests

There are no automated tests for this change, which is expected — it's infrastructure/deployment configuration (Dockerfile, shell scripts, PowerShell). Unit testing these would require Docker-in-Docker or extensive mocking. Manual testing against an actual deployment is the appropriate validation path.

However, the migration logic in `start-local.ps1` is complex enough that a brief documented test plan (or a comment in the script describing the expected one-shot behavior) would be valuable.

## Strong Concerns

**The `&&` to `||` fix in the GitHub auth validation is correct and important.** The original `&&` meant the check only failed when *all three* App credentials were empty simultaneously. With `||`, it correctly fails when *any* of the three is missing (and `GH_TOKEN` is also empty). This was a real logic bug. However, the fix was not called out in the commit messages I can see, which makes it easy to miss during rollback triage.

**The conditional forwarding of GitHub App env vars (redeploy.sh:49) is a good defensive change.** Only forwarding the three App credentials when all three are present prevents the container from receiving a partial/broken App auth config.

**The core design — moving HOME onto the persistent volume — is sound.** It's the simplest approach to persist all CLI auth state without per-tool special-casing.

## Confidence

**Medium-high.** The overall approach is correct and the individual changes are mostly well-reasoned. The blocking migration issue (#1) is a real data-loss risk on repeated local restarts. The hardcoded UID (#2) is a latent correctness bug that will bite on a base image update. The remaining items are polish. With those two fixed, this is ready to merge.
```
